# 登录系统架构升级设计文档

> **版本**: v1.0  
> **时间**: 2025 年 6 月  
> **范围**: 前台用户端（blog-client）+ 后端认证服务（blog-service）  
> **定位**: P0 + P1 级别生产安全加固，为后续 P2/P3 迭代奠定基础

## 一、改造背景与问题诊断

### 1.1 原始架构痛点

| # | 问题 | 严重等级 | 影响范围 |
|---|------|---------|---------|
| 1 | `Navbar.tsx` 和 `SidebarWidget.tsx` 各自维护一套完全重复的 `initAuth()` / `handleLogout()` / `user state`（约 100 行重复代码） | P1 | 前端可维护性 |
| 2 | `http-client.ts` 不自动注入 Token，各组件手动拼接 `Authorization` 头 | P0 | 安全遗漏风险 |
| 3 | Token 过期后前端无感知，用户操作静默失败 | P0 | 用户体验 |

| 7 | API Base URL 在 6 个文件中硬编码重复，环境切换易出错 | P1 | 部署灵活性 |

## 三、详细设计

### 3.1 Task 5：JWT 密钥生产安全校验

#### 设计动机

生产环境最常见的安全事故之一是 JWT 签名密钥使用默认值或过短密钥，导致 Token 可被伪造。通过在应用启动时增加 `@PostConstruct` 校验，可以在部署阶段就暴露问题，而非等到被攻击时才发现。

#### 核心逻辑

```java
@PostConstruct
public void validateSecret() {
    // 1. 检测是否仍使用默认弱密钥 → 打印 WARN 日志
    if (DEFAULT_SECRET.equals(secret)) {
        log.warn("【安全警告】JWT 签名密钥仍在使用默认值...");
    }
    // 2. 检测密钥长度是否 >= 32 字节（HS256 最低要求）
    //    不足则直接抛出异常，阻止应用启动
    if (secretLength < 32) {
        throw new IllegalStateException("JWT 密钥长度不足...");
    }
}
```

#### 设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 默认密钥检测 | WARN 而非阻断 | 开发环境需要便利性，仅告警不阻断 |
| 长度不足检测 | 直接抛异常阻断 | HS256 硬性要求，密钥过短 Token 签名不安全 |
| 校验时机 | `@PostConstruct` | Spring Bean 初始化阶段，最早的可用时机 |

### 3.2 Task 1：全局 AuthProvider 集中管理登录态

#### 设计动机

改造前 `Navbar` 和 `SidebarWidget` 各自维护完全相同的认证逻辑（`initAuth` / `handleLogout` / `user state` / `auth-change` 事件监听），共约 100 行重复代码。任何认证行为的变更都需要修改两个文件，极易遗漏导致状态不一致。

#### 架构设计

```
                    ┌──────────────────┐
                    │   AuthProvider    │
                    │  (React Context)  │
                    │                  │
                    │  user: UserInfo   │
                    │  isLoggedIn       │
                    │  isLoading        │
                    │                  │
                    │  login()          │
                    │  logout()         │
                    │  refreshUser()    │
                    └────────┬─────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
     useAuth() ▼    useAuth() ▼    useAuth() ▼
       Navbar      SidebarWidget   LoginModal
```

#### 核心 API 设计

```typescript
interface AuthContextType {
  user: UserInfo | null        // 当前登录用户信息
  isLoggedIn: boolean          // 是否已登录（计算属性）
  isLoading: boolean           // 初始化加载中（SSR 水合防闪烁）
  login: (token: string, userInfo: UserInfo) => void
  logout: () => void
  refreshUser: (info: Partial<UserInfo>) => void  // 局部更新（如头像）
}

interface UserInfo {
  nickname: string
  avatarUrl?: string | null
  username?: string | null
  email?: string | null
}
```

#### 状态同步机制

```
用户操作           localStorage         AuthProvider           UI 组件
   │                   │                    │                   │
   ├── login() ──────► │ setItem()          │                   │
   │                   │────────────────────► setState(user)     │
   │                   │                    │──► dispatch event  │
   │                   │                    │                   │
   ├── logout() ──────►│ removeItem()       │                   │
   │                   │────────────────────► setState(null)     │
   │                   │                    │──► dispatch event  │
   │                   │                    │                   │
   │    http-client    │                    │                   │
   │    401 响应 ──────► removeItem()       │                   │
   │                   │──► dispatch event──► 监听到 ──► re-sync │
   │                   │                    │                   │
   │  手动刷新页面 ─────►                    │                   │
   │                   │──► useEffect ──────► readFromStorage() │
```

#### 关键设计点

1. **`auth-change` 事件广播**：任何登录态变更（登录/登出/401过期）均通过 `window.dispatchEvent(new Event('auth-change'))` 广播，AuthProvider 内部监听并自动同步状态，确保跨 Tab 或跨组件状态一致

2. **SSR 安全**：`readUserFromStorage()` 内部使用 `typeof window === 'undefined'` 守卫，服务端渲染时返回 `null`，避免 SSR 报错

3. **脏数据保护**：`readUserFromStorage()` 对 `JSON.parse` 异常进行 try-catch，localStorage 中存储了非法 JSON 时不会崩溃，而是清除脏数据并返回 `null`

4. **`refreshUser()` 局部更新**：头像上传成功后无需重新登录，直接调用 `refreshUser({ avatarUrl: newUrl })` 局部更新 Context 中的 user 对象和 localStorage

### 3.3 Task 2：http-client 自动注入 Token + 401 拦截

#### 设计动机

改造前，Token 注入散落在各组件中：
- `article/[slug]` 手动 `headers: { Authorization: \`Bearer ${token}\` }`
- `CommentForm` 无 Token（评论是公开接口，但已登录用户的 Token 也应自动携带）
- `CommentSection` 无 Token

任何新增的 API 调用都需要手动记得拼接 Token，遗漏即导致鉴权失败。

#### Token 自动注入流程

```
request() 被调用
    │
    ├── 检查 headers 中是否已有 Authorization？
    │   ├── 是 → 跳过（尊重调用方手动指定的场景）
    │   └── 否 ↓
    │
    ├── typeof window === 'undefined'？（SSR 环境）
    │   ├── 是 → 跳过（无 localStorage）
    │   └── 否 ↓
    │
    ├── localStorage.getItem('token')
    │   ├── 有值 → headers['Authorization'] = `Bearer ${token}`
    │   └── null → 跳过（未登录，正常发请求）
    │
    └── 执行 fetch()
```

#### 401 自动拦截流程

```
fetch() 返回响应
    │
    ├── response.status === 401？
    │   ├── 否 → 正常处理
    │   └── 是 ↓
    │
    ├── handle401():
    │   ├── localStorage.removeItem('token')
    │   ├── localStorage.removeItem('user_info')
    │   └── window.dispatchEvent(new Event('auth-change'))
    │                           │
    │                           ▼
    │              AuthProvider 监听到事件
    │              → setState(null)
    │              → Navbar/SidebarWidget 自动切换为未登录 UI
    │              → 用户感知到"登录已过期"
    │
    └── 抛出 AppError（上层 catch 处理具体错误提示）
```

#### 迁移清单

| 文件 | 改动前 | 改动后 |
|------|--------|--------|
| `article/[slug]/page.tsx` | `fetch(API_BASE + '/articles/...')` + 手动 Token | `post('/articles/.../like')` |
| `article/[slug]/page.tsx` | `fetch(API_BASE + '/articles/...')` + 手动 header | `get('/articles/...')` |
| `CommentForm.tsx` | `fetch(API_BASE + '/articles/.../comments')` + 手动 Content-Type | `post('/articles/.../comments', body)` |
| `CommentSection.tsx` | `fetch(API_BASE + '/articles/.../comments')` | `get('/articles/.../comments')` |
| `CommentSection.tsx` | `fetch(API_BASE + '/comments/.../like')` | `post('/comments/.../like')` |

#### 设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| Token 存储位置 | localStorage | 现有方案，本次不改变（P3 阶段迁移 httpOnly Cookie） |
| SSR 兼容 | `typeof window` 守卫 | Next.js 服务端渲染时不存在 localStorage |
| 401 Toast 提示 | 由 AuthProvider 状态变更驱动 | 不在 http-client 中直接弹 Toast，保持请求层纯净 |
| 未手动指定 Auth 头时才注入 | 条件判断 | 保留特殊场景下手动覆盖 Token 的能力 |

### 3.4 Task 3：JWT Claims 增强

#### 设计动机

原始 JWT 仅存 `username`（subject），后端 Filter 解析 Token 后仅知道用户名，要获取 `userId` 和 `role` 仍需查库。在以下场景中造成不必要的 DB 查询：
- Controller 层需要 `userId` 关联数据（如评论点赞记录属于哪个用户）
- 细粒度权限控制需要 `role`（如 ADMIN 才能执行的操作）

#### Token 载荷对比

**改造前：**
```json
{
  "sub": "butvan",
  "iat": 1719000000,
  "exp": 1719604800
}
```

**改造后：**
```json
{
  "sub": "butvan",
  "userId": 1,
  "role": "ADMIN",
  "iat": 1719000000,
  "exp": 1719604800
}
```

#### 改动链路

```
AuthServiceImpl.login()
    │
    │  登录校验通过
    ▼
jwtUtil.generateToken(user.getId(), user.getUsername(), user.getRole())
    │
    │  写入 claims: { userId: 1, role: "ADMIN" }
    ▼
签发增强 Token 返回给前端
    │
    │  后续请求携带此 Token
    ▼
JwtAuthFilter.doFilterInternal()
    │
    ├── jwtUtil.getUsernameFromToken(jwt)    → "butvan"
    ├── jwtUtil.getUserIdFromToken(jwt)      → 1
    ├── jwtUtil.getRoleFromToken(jwt)        → "ADMIN"
    │
    └── 存入 Authentication.details:
        {
          "userId": 1,
          "role": "ADMIN",
          "remoteAddress": WebAuthenticationDetails
        }
    │
    ▼
Controller 层通过 SecurityContext 获取：
    Authentication auth = SecurityContextHolder.getContext().getAuthentication();
    Map<String, Object> details = (Map<String, Object>) auth.getDetails();
    Long userId = (Long) details.get("userId");
    String role = (String) details.get("role");
```

#### 向后兼容

保留了原始 `generateToken(String username)` 方法，已签发的旧 Token 仍可正常解析（`getUserIdFromToken` 对旧 Token 返回 `null`），实现**平滑升级**，无需强制用户重新登录。

### 3.5 Task 4：登录接口 Redis 频率限制

#### 设计动机

登录接口是暴力枚举密码的主要攻击面。利用项目已有的 Redis 基础设施（`StringRedisTemplate`），实现基于滑动窗口的频率限制，在不影响正常用户体验的前提下有效防御暴力破解。

#### 限流策略

```
                    滑动窗口：5 分钟
                    ┌──────────────────────┐
用户/IP 登录请求 ──►│  失败计数 < 5 → 放行   │──► AuthService.login()
                    │  失败计数 >= 5 → 429  │──► "操作过于频繁"
                    └──────────┬───────────┘
                               │
                    ┌──────────▼───────────┐
                    │   登录失败            │
                    │   → INCR user key    │
                    │   → INCR ip key      │
                    │   → 首次设置 TTL 5min │
                    └──────────────────────┘

                    ┌──────────────────────┐
                    │   登录成功            │
                    │   → DELETE user key  │  ← 立即清零
                    │   → DELETE ip key    │
                    └──────────────────────┘
```

#### Redis Key 设计

| Key 模式 | 示例 | 用途 | TTL |
|----------|------|------|-----|
| `login:fail:user:{username}` | `login:fail:user:butvan` | 按用户名限流 | 5 分钟 |
| `login:fail:ip:{clientIp}` | `login:fail:ip:192.168.1.100` | 按 IP 限流 | 5 分钟 |

#### 双维度限流原因

| 维度 | 防御场景 |
|------|---------|
| 用户名 | 防止针对单一账号的密码枚举 |
| IP | 防止攻击者使用不同用户名批量尝试（字典攻击） |

#### 核心代码逻辑

```java
// 1. 递增计数器（原子操作）
Long count = stringRedisTemplate.opsForValue().increment(key);
if (count != null && count == 1L) {
    // 首次写入 → 设置窗口过期时间
    stringRedisTemplate.expire(key, 5, TimeUnit.MINUTES);
}

// 2. 前置校验
Long userFails = getCount("login:fail:user:" + username);
if (userFails >= 5) {
    long ttl = getRemainingTtl(key);
    throw new BusinessException(429, "操作过于频繁，请 X 分钟后重试");
}

// 3. 成功后清除
stringRedisTemplate.delete("login:fail:user:" + username);
stringRedisTemplate.delete("login:fail:ip:" + clientIp);
```

#### Controller 层集成

```java
@PostMapping("/login")
public Result<LoginVO> login(LoginDTO loginDTO, HttpServletRequest request) {
    String clientIp = extractClientIp(request);  // 兼容 Nginx X-Forwarded-For

    // 1. 前置频率校验（超限直接抛 429）
    loginRateLimiter.checkRateLimit(loginDTO.getUsername(), clientIp);

    try {
        LoginVO loginVO = authService.login(loginDTO);
        // 2. 成功清零
        loginRateLimiter.clearFailCount(loginDTO.getUsername(), clientIp);
        return Result.success(loginVO);
    } catch (BusinessException e) {
        // 3. 失败计数（429 本身不计数）
        if (e.getCode() != 429) {
            loginRateLimiter.recordFailure(loginDTO.getUsername(), clientIp);
        }
        throw e;
    }
}
```

#### IP 提取策略

```java
private String extractClientIp(HttpServletRequest request) {
    // 优先级：X-Forwarded-For > X-Real-IP > remoteAddr
    // 兼容 Nginx / CDN 反向代理场景
    String ip = request.getHeader("X-Forwarded-For");
    if (valid(ip)) return ip.split(",")[0].trim();  // 多级代理取第一个
    ip = request.getHeader("X-Real-IP");
    if (valid(ip)) return ip.trim();
    return request.getRemoteAddr();
}
```

## 四、安全分析

### 4.1 改造前后安全对比

| 安全维度 | 改造前 | 改造后 |
|---------|--------|--------|
| Token 注入 | 各组件手动拼接，易遗漏 | http-client 自动注入，零遗漏 |
| Token 过期 | 用户无感知，操作静默失败 | 401 自动清除登录态 + UI 同步切换 |
| Token 载荷 | 仅 username，鉴权后仍需查库 | userId + role，Controller 可直接使用 |
| 暴力破解 | 无任何防护 | 5分钟/5次 滑动窗口限流 |
| 密钥安全 | 默认弱密钥无告警 | 启动时校验 + 长度不足直接阻断 |
| JWT 签名 | HS256 + 84字符密钥 | 不变（已满足安全要求） |

### 4.2 已知局限与后续规划

| 级别 | 内容 | 建议时机 |
|------|------|---------|
| P2 | Access Token + Refresh Token 双 Token 机制 | 当 Token 有效期需要缩短时（如 15 分钟） |
| P2 | Redis Token 黑名单 | 需要支持"踢人下线"或主动吊销 Token 时 |
| P3 | Token 改为 httpOnly Cookie 存储 | 需要防御 XSS 窃取 Token 时 |
| P3 | CSRF Token 防护 | 配合 httpOnly Cookie 一起实施 |
