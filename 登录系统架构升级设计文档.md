# 登录系统架构升级设计文档

> **版本**: v2.0  
> **时间**: 2025 年 6 月  
> **范围**: 前台用户端（blog-client）+ 后端认证服务（blog-service）

---

## 一、改造概述

本次升级分两阶段完成，目标是将登录系统从"能用"提升到"生产级安全"。

### 第一阶段：基础架构加固（5 项）

| 改造项 | 说明 |
|--------|------|
| AuthProvider 全局状态 | 消除 Navbar/SidebarWidget 等组件中约 100 行重复认证代码，统一由 `useAuth()` Hook 管理 |
| http-client 统一请求层 | 所有 API 调用走 `get()`/`post()`/`upload()`，消除散落在各组件中的手动 `fetch` |
| JWT Claims 增强 | Token 载荷增加 `userId` + `role`，减少不必要的数据库查询 |
| 登录频率限制 | Redis 滑动窗口限流（5 分钟 / 5 次），双维度（用户名 + IP）防暴力破解 |
| JWT 密钥安全校验 | 启动时检测默认弱密钥和长度不足，阻止不安全的 Token 签名 |

### 第二阶段：httpOnly Cookie + 双 Token 机制

| 改造项 | 说明 |
|--------|------|
| 双 Token 签发 | Access Token（15 分钟）+ Refresh Token（7 天），缩小单 Token 泄露的时间窗口 |
| httpOnly Cookie 存储 | Token 不再存入 localStorage，改为 httpOnly Cookie，JS 无法读取，彻底防御 XSS 窃取 |
| 401 自动续期 | Access Token 过期时 http-client 自动调用 `/auth/refresh` 换取新 Token 并重试，用户无感知 |
| Refresh Token 黑名单 | 登出时将 Refresh Token 的 jti 写入 Redis 黑名单，支持主动吊销 |

```
│                                                         │
│  LoginRateLimiter → Redis 滑动窗口限流                   │
└─────────────────────────────────────────────────────────┘
```

## 三、双 Token 机制详解

### 3.1 Token 设计

| 属性 | Access Token | Refresh Token |
|------|-------------|---------------|
| 有效期 | 15 分钟 | 7 天 |
| 载荷 | userId + username + role + type | userId + username + type + jti |
| 存储方式 | httpOnly Cookie（Path=/api） | httpOnly Cookie（Path=/api/auth/refresh） |
| 用途 | 每次 API 请求鉴权 | 仅用于换取新 Access Token |

### 3.2 请求流程

```
正常请求：
  浏览器 → Cookie(access_token) → JwtAuthFilter → 鉴权通过 → Controller

Token 过期（用户无感知）：
  浏览器 → Cookie(过期) → 401
    → http-client 自动 POST /auth/refresh（Cookie 携带 refresh_token）
    → 后端签发新 access_token → Set-Cookie
    → http-client 重试原始请求 → 成功

Refresh Token 也过期：
  → /auth/refresh 返回 401
  → http-client 清除 localStorage(user_info) + 广播 auth-change
  → AuthProvider 状态置空 → UI 切换为未登录
```

### 3.3 登出流程

```
用户点击退出
  → POST /auth/logout（Cookie 携带 refresh_token）
  → 后端将 jti 写入 Redis 黑名单（TTL = Token 剩余有效期）
  → 清除 access_token + refresh_token Cookie
  → 前端清除 localStorage(user_info) + 广播 auth-change
```

## 四、安全对比

| 维度 | 改造前 | 改造后 |
|------|--------|--------|
| Token 存储 | localStorage（XSS 可直接窃取） | httpOnly Cookie（JS 无法读取） |
| Token 有效期 | 7 天单 Token | 15 分钟 Access + 7 天 Refresh |
| Token 过期处理 | 用户无感知，操作静默失败 | 自动续期，续期失败才提示重新登录 |
| 登出效力 | 仅清除前端，Token 仍有效 | Redis 黑名单吊销，服务端真正失效 |
| 暴力破解 | 无防护 | 5 分钟 / 5 次滑动窗口限流 |
| 密钥安全 | 默认弱密钥无告警 | 启动校验 + 长度不足直接阻断 |
| CSRF 防护 | 无（localStorage 不需要） | SameSite=Lax Cookie 提供基础防护 |

## 五、后台管理端兼容性

本次改造**仅影响前台用户端（blog-client）**。后台管理端（blog-admin）保持原有方案不变：

- JwtAuthFilter 同时支持 Cookie 和 Authorization 头，后台继续使用 `Authorization: Bearer <token>` 方式
- `AuthServiceImpl.login()` 仍在响应体中返回 token 字段，后台正常读取
- SecurityConfig 放行的公开接口不受影响
