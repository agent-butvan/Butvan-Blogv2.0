## 🏗️ 架构四大交互层级 (Swimlanes)
架构图由左至右分为四个系统物理层（泳道），体现了清晰的边界和职责分离：
1. **客户端 (Browser / Client)**：处理用户的交互、扫码状态监听、通过 WebSocket 实时更新 UI 并完成 Token 交换登录。
2. **微信平台 (WeChat Platform)**：处理用户微信端的扫码关注、发送文本等动作，并将事件（XML）推送至后端。
3. **后端服务 (blog-service)**：核心业务控制器（`WeiXinEventServiceImpl`），实现事件决策路由、绑定逻辑和交换码分发。
4. **缓存与数据库 (Redis / DB)**：存放临时状态（`ws_id` 映射、交换码、120s 临时绑定关系）以及持久化用户数据（`WechatUser` & `User`）。

## 🔄 核心业务流转四大阶段
架构图细致刻画了以下四个生命周期：

### 阶段一：建立连接与二维码初始化
*   用户进入登录页面，前端建立 **WebSocket** 连接（持有 `ws_id`）。
*   前端请求二维码，后端生成带唯一 `ticket` 参数的专属二维码，并在 Redis 缓存中维护 `ticket -> ws_id` 的绑定。

### 阶段二：用户扫码与微信事件分流
用户扫码后，微信服务器进行逻辑决策并回调推送：
*   **若用户未关注** ➔ 触发并推送 `subscribe` (关注) 事件。
*   **若用户已关注** ➔ 触发并推送 `SCAN` (扫码) 事件。

### 阶段三：后端核心逻辑与校验 (分支流转)
后端在 `WeiXinEventServiceImpl` 中分流处理：
*   **`userFirstRegister` (关注分支)**:
    1.  **名额防超限校验**：统计当前已关注用户总数，若 `>= 20` 名额已满直接拦截。
    2.  **扫码通知**：通过 `ticket` 获取 `ws_id`，WS 实时通知前端已扫码。
    3.  **临时绑定**：在 Redis 缓存 `open_id -> ticket` 以关联后续回复。
    4.  **二次关注自动登录**：判断若已绑定过账号且 `status=0`，一键重置为 `1` 并生成交换码，WS 通知登录成功。
    5.  **首次关注引导**：发送微信模板消息，提示用户发送邮箱以完成注册绑定。
*   **`userLogin` (已关注扫码分支)**:
    1.  检测是否已绑定系统账号（`userId`）。未绑定则抛出异常提示重试。
    2.  检测系统用户邮箱是否存在。若不存在，WS 发送邮箱异常消息。
    3.  登录成功：更新最后登录时间，生成交换码，WS 通知登录并发送成功消息。

### 阶段四：回复邮箱文本与注册绑定 (`text` 事件)
*   未绑定用户在微信内回复邮箱 ➔ 后端触发 `text` 事件提取并校验邮箱格式。
*   根据 `open_id` 查询 Redis 中暂存的 `ticket` 获取 `ws_id`。
*   **若邮箱账号不存在 (全新用户)** ➔ 创建新 User 并与 `WechatUser` 绑定，WS 通知注册并自动登录。
*   **若邮箱账号已存在 (仅绑定微信)** ➔ 将已有 User 的 ID 关联到 `WechatUser.userId`，完成绑定，WS 通知登录。
